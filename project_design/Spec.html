1. Mental model

cmux is built on three nested concepts:

  1. Project — a top-level unit; corresponds 1:1 with a repo + optional service config. Each project has a user-assigned name and color. Represented as a browser-style tab at the top of the window.
  2. Workspace — a working context within a project: typically a git branch/worktree, a task, or an experiment. A project has 1..N workspaces. Listed in the sidebar of the active project.
  3. Pane — a running agent or shell inside a workspace. Multi-pane splits live in the main content area. Each pane hosts a tab strip of its own (multiple agents/shells can share a pane slot). Analogous to iTerm/tmux panes but with tabs.
The single most important rule: the tab bar represents projects, the sidebar represents workspaces within the active project, and panes represent agents/shells within the active workspace. Each layer does exactly one job.

Why tab-per-project (not one-sidebar-with-many-projects)

  • The common case — one workspace per project — produces a clean single-row sidebar with no grouping scaffolding.
  • Projects already feel like "separate worlds" (separate repos, separate services, separate terminal environments). Tabs reinforce that.
  • Drag-out-to-new-window works like browsers: gives power users window-per-project without forcing it.
  • Multi-level badges roll up honestly: pane → workspace row → project tab.

5. Components

5.1 · Project tab

  • Anatomy: 8×8px rounded color square · project name · optional signal pips · × close.
  • Active: filled with bg1, 2px top border in project color, 1px L/R borders in line1, no bottom border (merges into sidebar).
  • Inactive: transparent background, no top border, bottom border in line1.
  • Close: × in fg3, visible on hover (or always on active).
  • Pips: 6px yellow dot = project has a workspace needing input; 6px red dot = a service has errored. Both shown on the right side, before the ×.
  • Drag-out: holding and dragging a tab outside the window spawns a new window with only that tab; reversible by dragging back.

5.2 · Sidebar

  • Repo row at top: branch icon + repo slug in mono, fg2. 10px vertical padding. Non-interactive (just info).
  • Workspaces header: uppercase eyebrow + count + +. Clicking + opens "new workspace" palette.
  • Workspace row (see 5.3).
  • Services footer (optional): uppercase eyebrow + count + +. One row per service: kind icon · title · status pill.

5.3 · Workspace row (aggregated)

  • Leading dot (6px) — priority-highest-wins: red (error) → yellow (needs input) → blue (new output) → green (running) → gray (idle). Pulses iff any actionable signal is unseen.
  • Title (12.5px, 500), single-line ellipsis.
  • Aggregated pips (trailing) — one chip per signal kind with a count. E.g. ⌁2 !1 ♪1.
  • Meta line: branch · ◆N agents · ›N shells · age.
  • Active: accentSoft background, 2px accent rail on the left edge.
  • Active + has panes: row expands inline to show one line per pane, each with its own specific signal dot + label. This is how 2+ agents in the same workspace get disambiguated.

5.4 · Tabbed pane

  • Tab strip on bg0 (matches app chrome). Tabs overlap the pane body's top border.
  • Pane body is bg2 with a 1px line1 border, 4px radius.
  • Tab chrome: 28px tall · 6px status dot · label (JetBrains Mono 11) · × close. Active tab has 2px top border in accent + merges into pane body. Inactive tabs have line1 bottom border.
  • + button at end of strip: opens a mini-palette ("new agent · new shell · new file").
  • No pulsing pane body. Ever. The signal is always on the tab's leading dot.
  • Split: panes are arranged in a splittable grid (top-row 1, bottom-row 2 is default on first split). User can drag dividers; Cmd+D horizontal split, Cmd+Shift+D vertical.

6. Notifications

Three-tier signal language. Intensity matches stakes.

KindTierColorBehaviorSource
needs input Actionableyellow Dot pulses when unseen. Static when seen (user has looked at pane but not responded). Clears when resolved. Agent emits a "waiting for input" event.
error Actionablered Same as needs-input. Stronger priority — red wins over yellow in roll-ups. Non-zero exit, crash, or agent reports failure.
new output Informationalaccent blue Static dot (no pulse). Clears when the user focuses that pane. Any unread content since last focus.
bell Ambientorange Static dot (no pulse). Auto-clears after 5 seconds OR on pane focus, whichever first. Terminal BEL (\a) from shell.
running Stategreen Static dot. Not a "notification" — just a live status. Process/agent is active.
idle Stategray Static dot. Quietest. No active process.

Lifecycle

  1. Emit: agent/shell produces an event. Signal attaches to the pane tab.
  2. Roll up: the workspace row's aggregated pip count increments; the project tab's roll-up pip lights up iff this is actionable.
  3. Pulse: if actionable and unseen, the pane-tab dot pulses. Pulse decays after 8 seconds regardless — attention fatigue is real.
  4. Seen: when the user focuses the pane (or even just the workspace), the pulse stops. Dot remains colored until resolved.
  5. Resolved: user responds to the agent / dismisses the error / clears the bell. Dot disappears. Roll-ups decrement.
No pulsing borders. The existing blue pulsing pane border is replaced entirely by tab-dot signals. A pulsing border competes with focus state and overwhelms the pane content.

Roll-up priority

When a single dot must represent multiple signals (workspace row leading dot, project tab pip), higher priority wins:

error (red) > needs-input (yellow) > new-output (blue) > running (green) > bell (orange) > idle (gray)

Aggregated pip chips (trailing, with counts) are shown in addition to the leading dot — so you see both the top-priority state and the breakdown.

8. State & persistence

  • Project list: persisted globally (across windows). Each has { id, name, color, repoPath, serviceConfig, lastOpenedAt }.
  • Window state: persisted per-window — which project tabs are open, which is active, sidebar width, split geometry per workspace.
  • Workspace state: persisted per-workspace — pane layout, per-pane tab list, scrollback, agent conversation state.
  • Signals: in-memory per session. "Seen" state is per user-visit to the app (resets on relaunch). Bells clear on close.
  • Services: keep running even when no workspace in the project is open. Stopping services is an explicit user action.
  • Closing a project tab: does NOT stop services or kill agents. Use "Quit project" menu item for that.

9. Edge cases

Empty project

Project tab with zero workspaces: sidebar shows the repo row + services (if any), and the main content area shows an empty-state card with "New workspace" CTA.

Many workspaces

Sidebar workspace list is scrollable. No artificial cap. If >15 workspaces, show a search input at the top of the list (focused via ⌘F while sidebar is focused).

Many projects

When project tabs overflow, show a chevron "more" menu at the right of the tab strip (same pattern as browsers). Never shrink tabs below 100px; always scroll.

Same directory, multiple agents

This is explicitly supported and was a motivating case. Each agent lives in its own pane tab. The sidebar workspace row aggregates "2 agents need input" rather than showing two identical rows. The expanded per-pane detail in the active row disambiguates.

Drag-out to new window

A single-tab window still shows the tab strip (so the gesture is reversible). Dragging the last tab back to an existing window merges them.

10. Open questions (flag for the coding agent)

  • Pulse duration: spec says 8 seconds. Needs to feel right — test and tune.
  • Bell auto-clear: 5s is arbitrary. Consider configurable per-user.
  • Sidebar search: threshold of 15 workspaces is a guess. May be better to always show the input, ghost-faded, and activate on typing.
  • Expanded workspace pane list: currently only active workspace expands. Consider letting the user pin-expand others.
  • macOS idioms: the tab bar is custom (mimics Arc/Chrome rather than NSTabBar). Decide whether to adopt native NSWindowTab or keep the custom treatment for brand identity.
  • Accessibility: all color-coded signals need a secondary encoding (icon or text) for colorblind users. The dot + text-pip combination starts to address this; audit fully.
  • Status-bar / Dock integration: when the app is backgrounded, actionable signals should surface via notification center and/or the Dock badge. Spec that separately.
